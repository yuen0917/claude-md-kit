# Prompt:請 Fable 5 產生「通用 CLAUDE.md 範本與專屬 Workflow」

> 使用方式:將以下全部內容貼給 Fable 5 即可。若你有特定專案類型(如研究、資料分析、後端服務),可在第二節「背景與目標」補上一句說明,產出會更貼合。

---

## 一、你的角色與任務

你是一位熟悉 Claude Code(agentic coding)與 AI agent context 工程的資深顧問。我需要你設計一套「可套用於任何專案」的 CLAUDE.md 範本,以及一份配套的 agent workflow。目標是讓任何團隊或個人都能以此為起點,快速建立一份精簡、有效、能實際改變 agent 行為的專案設定,並讓 AI agent 能更可靠地管理專案、逐步達成目標。

請注意:這不是要你為某個特定專案填好內容,而是要你產出一套「範本 + 準則 + 一個填好的範例」,讓使用者能依自己的專案套用。

## 二、背景與目標

- 這份範本會被不同專案重複使用,因此通用規則與專案專屬規則必須清楚分層。
- 產出應內建「共同可遵守的標準」,同時保留讓各專案填寫差異的彈性。
- 最終目的是提升 agent 管理專案、達成目標的可靠度,而非堆砌規則。

（若使用者已補充特定專案類型,請據此調整範例與慣例段落。）

## 三、交付物(請完整產出以下項目)

1. **通用 CLAUDE.md 範本**:含清楚標註的 placeholder(例如 `<專案名稱>`、`<build 指令>`),並在每個區塊附一至兩句註解,說明「這裡該填什麼、為什麼」。
2. **配套 Workflow 文件**:描述 agent 從接到任務到完成的標準流程,需可跨 session 使用。
3. **分層放置指引**:說明哪些內容應放全域設定、哪些放專案根目錄、哪些應以路徑作用域(rules)方式載入。
4. **一個填好的範例**:以一個假想專案示範範本套用後的實際樣貌,供使用者對照。

## 四、必須遵守的設計原則(硬性要求)

以下原則來自社群與實務驗證,請務必內化到產出中,而非僅條列出來。

### 精簡優先
- CLAUDE.md 每個 session 都會載入 context,與實際任務搶注意力。根目錄的 CLAUDE.md 請控制在 60 至 80 行以內;超過此長度,agent 會開始忽略部分內容。
- 對每一行套用這個判斷準則:「如果刪掉這行,agent 會不會因此出錯?」若不會,就不該寫。
- 對少數關鍵且不可違反的規則,可用 IMPORTANT 或 YOU MUST 標記以提高遵循率,但須節制,濫用會稀釋效果。

### 分層與漸進式揭露
- 通用、跨專案共用的行為準則應放在全域設定,讓每個專案的 CLAUDE.md 只保留專案專屬內容,避免重複。
- 對較大的 codebase,支援在子目錄放置 CLAUDE.md(僅在該目錄工作時載入),或以 `.claude/rules/` 搭配路徑 glob,讓特定規則只在特定檔案類型或目錄啟用。
- 不要用 @-mention 直接嵌入整份文件(會在每個 session 佔用 context);改為告訴 agent「何時該去讀哪份文件」。

### 內容取捨
明確區分兩層內容。

通用行為準則(跨專案):先讀再改、改完必驗(將「完成」定義為通過測試／type check／lint／實際執行)、最小改動、遇 blocker 就停並浮現讓人決定、不確定就查證不編造、破壞性或不可逆操作先確認。

專案專屬內容:專案目的與領域脈絡、技術棧與環境、驗證用的具體指令(build／run／test／lint)、高層架構地圖、慣例與陷阱、外部資源位置。

同時明確排除以下浪費 context 的內容:
- 人格指令(如「當個資深工程師」「一步步思考」)——模型本身已有強系統層指令。
- 工具已能強制執行的格式規則——若已設定 formatter,改用 hook 處理,而非寫進 CLAUDE.md。
- 從專案結構一看即知、無法轉為具體行動的資訊。

### Workflow 骨架
Workflow 請以這個經多個熱門實作收斂出的模式為主軸:Research → Plan → Execute → Review → Ship(定位 → 規劃 → 執行 → 驗證 → 交付)。至少涵蓋:
- 定位:讀取 CLAUDE.md 與相關檔案,將任務目標以自己的話覆述並確認理解無誤。
- 規劃:非瑣碎任務先列步驟,必要時先與人對齊再動手;避免過度工程化成一長串自訂指令。
- 執行:一次一個邏輯單元,保持 diff 聚焦。
- 驗證:實際執行前面定義的檢查,真正確認過才算完成。
- 交付:精簡說明改了什麼、為什麼、以及未解決事項;採小而原子的 commit。

### 跨 session 狀態管理
- 為長期專案設計一個單一事實來源(例如一份 knowledge／decisions 檔):任務開始前先讀取相關脈絡、架構、狀態與決策;完成後更新記錄變更、狀態、決策與學到的事。
- 說明如何與 Claude Code 內建的自動記憶機制分工——有些內容 agent 會自行學會,不必寫進 CLAUDE.md。

### MCP 與 Skill 的取捨
- MCP 有 context 成本:整體 MCP 佔用若超過約 20k token,會嚴重壓縮可用於實際工作的空間。請在 workflow 或指引中強調 MCP 需精選、按需啟用。
- 優先考慮 Skill:Skill 採漸進式揭露,掃描階段每個僅約 100 token,啟用時才載入完整內容(通常低於 5k token),且可跨工具攜帶。

### 安全
- 提醒使用者:外部內容(如抓取的網頁、論文、第三方檔案)屬不受信任輸入,可能夾帶對抗性指令,處理時應收緊工具權限。
- 提醒 Skill 與 MCP 可執行任意程式碼,只安裝來自可信來源者,安裝前先審視其內容。
- 秘密(API key、憑證)絕不寫入 CLAUDE.md 或任何會被 agent 讀取或提交的設定檔;改用環境變數並列入忽略清單。

## 五、輸出格式要求

- 以繁體中文撰寫說明文字;CLAUDE.md 範本與範例本身可依慣例保留必要的英文技術詞與指令。
- 各交付物請以清楚的標題分段,範本與範例以程式碼區塊呈現,方便直接複製。
- 每個 CLAUDE.md 區塊請附一至兩句註解,說明其用途與填寫原則。
- 產出結尾請附一份「精簡檢查清單」,讓使用者套用範本後能自我檢核(例如:是否超過 80 行、是否含人格指令、是否有可驗證的指令、MCP 是否精選等)。

## 六、若有需要澄清之處

若在動手前,你判斷有任何影響產出品質的關鍵資訊不足,可先提出至多三個聚焦的問題再開始;否則請直接產出完整交付物,並在做出假設之處以註記標明。
